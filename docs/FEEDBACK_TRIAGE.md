# Feedback Triage Process

**The Goal**: Turn every user interaction into a high-leverage improvement to Code Kit Ultra's adoption curve.

---

## 1. Capture (The Inbox)
All feedback—GitHub issues, demo notes, or direct messages—must be logged in `docs/USER_FEEDBACK_LOG.md` within 24 hours.

## 2. Categorize
Assign every piece of feedback to one of these buckets:
- **Environment**: Setup or dependency issues.
- **Onboarding**: "First 10 minutes" friction.
- **Capability**: Missing features or skill gaps.
- **UX**: CLI confusion or report readability.
- **Positioning**: "I don't get what this is for."

## 3. Prioritize (The Impact/Effort Matrix)
- **Quick Wins**: High impact, low effort (e.g., README typos, better CLI error messages).
- **Hard Wins**: High impact, high effort (e.g., new adapter integrations).
- **Polishing**: Low impact, low effort.
- **Backlog**: Low impact, high effort.

## 4. Implement & Close Loop
When a fix is pushed:
1.  Update `docs/USER_FEEDBACK_LOG.md` in the "Resolved" section.
2.  Update `docs/TRACTION_DASHBOARD.md` metrics.
3.  Notify the original user (if possible).

---

*Code Kit Ultra: Relentless improvement through active listening.*
